把 AI 写代码，从会聊天，变成会接着干活

Trellis 不是另一个写代码模型，它更像是给 Claude Code、Codex 这些 AI 工具加上一套项目记忆、任务状态和工作流程。 当前 0.5.x 版本已经支持十多个平台，但这份页面先拿 Claude Code 和 Codex 做主线，因为它们最能看清 slash commands、skills、hooks、任务文件和 journal 的差别。 你可以只接 Claude、只接 Codex，或者两个都接；下面我会先讲框架，再讲单独使用和同时使用时该怎么操作，最后给一条从 0 到 1 的完整实战路径。

一句话理解

Claude Code / Codex 是干活的 AI，Trellis 是让 AI 在项目里知道"这是什么项目、当前任务到哪、上次做到哪、接下来该做什么"的那一层。

这页重点解决

1. 第一次接触 Trellis 到底该怎么理解。
2. 只用 Claude、只用 Codex、或者两个都接时分别怎么做。
3. 中断以后怎么知道项目进度，用户又该看哪里。

Overview

Trellis 到底是什么

如果只用网页版 AI，你每次都在重新开一个临时群聊。Trellis 的作用，是把项目真实上下文落在仓库里，让 AI 每次不是"重新猜"，而是"先读事实，再继续"。

01
没有 Trellis

AI 像聪明但健忘的临时工

你说过的技术栈、目录规则、任务边界、上次做到哪，很容易随着上下文清空或换会话被冲掉。 结果就是同一件事反复解释，甚至换个会话就写偏。

常见现象： AI 忘了项目规则、续工时串仓库、写完代码却没有留下可追踪的过程记录。
用了 Trellis

AI 像会看项目档案的正式协作者

Trellis 把项目共享入口、规范、任务、日志沉淀成文件。AI 开工前先读这些材料，收工时再把本次结果记回项目里。 这样你换模型、换会话、甚至换一天继续，都还有证据链。

核心价值： 把"AI 记住了什么"变成"仓库里确实写了什么"。
入口

项目是什么

用 AGENTS.md 这类跨平台入口文件告诉 AI 从哪里读项目事实。它不一定包含全部业务背景，真实规则还要落到 .trellis/spec/ 等文件里。

  • 这是哪个项目
  • 哪些目录最关键
  • 哪些规则是长期有效的
流程

当前任务是什么

用 .trellis/tasks/ 放任务、PRD、阶段产物，再通过 .trellis/.runtime/sessions/ 里的会话指针判断本次 AI 会话正在做哪一件事。 具体任务目录、状态和阶段由 task.py 管理，不再只看旧式 .current-task 文件。

  • 当前任务目录
  • 任务状态与目标
  • 已归档任务留作追溯
记忆

上次做到哪

用 .trellis/workspace/ 写 journal，让你和 AI 都能看见上个 session 做了什么、还差什么、下一步建议是什么。

  • 本次完成了什么
  • 留下了哪些文件
  • 下次应从哪接着做
Start

先分清三件事：接哪一个、首次接入、日常开工

这是最容易混淆的一步。Trellis 不要求你同一项目必须同时装 Claude 和 Codex。 trellis init 是把 Trellis 接进项目，不是每天上班的"启动按钮"。

02
只用 Claude

适合就想走 Claude Code 工作流

  • 初始化：trellis init -u 你的名字 --claude --skip-existing

  • 日常开工：claude；Claude hook 会注入上下文，显式续工用 /trellis:continue
  • 主要看：.claude/，按项目需要补读 AGENTS.md
只用 Codex

适合就想走 Codex skill 工作流

  • 初始化：trellis init -u 你的名字 --codex --skip-existing
  • 会生成：.trellis/、AGENTS.md、.codex/、.agents/skills/
  • 日常开工：codex；首次无任务时按提示用 $trellis-start，续工用 $trellis-continue
  • 主要看：AGENTS.md、.agents/skills/、.codex/
两个都接

适合同仓库切换 Claude 和 Codex

  • 初始化：trellis init -u 你的名字 --claude --codex --skip-existing
  • 会同时生成：.claude/、.codex/、.agents/skills/ 和共享层
  • 日常开工：今天用哪个工具，就启动哪个
  • 共享事实层：AGENTS.md + .trellis/；平台入口仍然各走各的
先判断依据

源码、生成文件、官方 latest 不一致时，先信谁？

优先级 看哪里 这次项目里的结论
1 当前项目实际生成文件 .trellis/.version 是 0.5.15；.claude/commands/trellis/ 只有 continue.md 和 finish-work.md。
2 本机 CLI 与官方 npm latest trellis --version 和 npm latest 都是 0.5.16；trellis update --dry-run 会告诉你升级会改哪些模板。
3 下载的旧源码目录 Trellis/package.json 是 0.3.5，可用来理解历史设计，但不能直接照抄里面的旧命令到 0.5.x 项目。
这次源码剖析后的教学口径：

Trellis/src/cli/index.ts 只能证明 CLI 暴露了 init / update 以及 --claude / --codex 这些平台开关；Trellis/src/configurators/claude.ts 和 Trellis/src/configurators/codex.ts 解释了模板目录怎么被复制到项目里。 但这份源码快照里的 src/templates/claude/commands/trellis/start.md、 record-session.md、Codex 旧 $start 文案都属于旧版模板线索。 真正教用户复制时，必须以本项目生成后的 .claude/、.codex/、 .agents/skills/ 和 .trellis/workflow.md 为准。

证据文件 教案里应该怎么讲
.trellis/workflow.md 最新日常流程是 Phase 1 Plan、Phase 2 Execute、Phase 3 Finish；finish-work 在工作提交之后。
.trellis/scripts/task.py create 进入 planning；start 进入 in_progress；finish 只清指针、主要用于排错；archive 才完成归档。
.claude/settings.json Claude 有 SessionStart、PreToolUse、UserPromptSubmit hooks，适合主会话派发 implement/check agent。
.codex/config.toml + .codex/hooks.json Codex 默认保留 AGENTS.md 为项目说明入口，hook 只注册 UserPromptSubmit。用户级开启 hooks、信任项目并在 /hooks 批准后，普通开发默认 inline。
.agents/skills/trellis-*/SKILL.md Codex 用户复制时写 $trellis-start、$trellis-continue、$trellis-before-dev、$trellis-check。
首次接入

只在第一次做

新项目第一次装 Trellis，或者老项目第一次补装 Claude / Codex 支持时，再用初始化命令。 trellis init 负责把平台目录、.trellis/ 和 AGENTS.md 写进项目。

initial setup 
# 只在第一次接入时运行
git init
npm install -g @mindfoldhq/trellis@latest

# 只用 Claude
trellis init -u 你的名字 --claude --skip-existing

# 只用 Codex
trellis init -u 你的名字 --codex --skip-existing

# 两个都接
trellis init -u 你的名字 --claude --codex --skip-existing
Mac 权限提示

如果执行 npm install -g @mindfoldhq/trellis@latest 报 permission denied，可以改用 sudo npm install -g @mindfoldhq/trellis@latest。输入的是你的开机密码；终端里打字时通常不会显示字符，这是正常现象，输完按回车即可。

日常开工

每天真正会用到的是这个

项目已经接入 Trellis 之后，日常只是进入项目目录，然后启动你要用的 AI 工具。 不要把 trellis init 当成每天开工按钮。

daily workflow 
# Claude Code
cd "你的项目路径"
claude

# Codex
cd "你的项目路径"
codex
Codex 首次使用前再做一次设置：

项目里的 .codex/config.toml 不能替你打开用户级 feature。你需要在 ~/.codex/config.toml 里启用 [features].hooks = true，把项目标记为 trusted，然后在 Codex 里用 /hooks 批准 Trellis hook。

最实用的判断标准： 如果你只是想"今天继续做昨天的项目"，不要再跑 trellis init。 你需要做的是启动这次准备使用的那个工具，例如 claude 或 codex，然后先让 AI 读取项目证据链。

03
必看 1

AGENTS.md：先确认项目入口

它更像跨平台入口和导航，不一定写满业务背景。你先看它，是为了知道 AI 应该从哪里继续读项目事实。

  • 适合开工前先看。
  • 适合续工时先确认"我是不是进对仓库了"。
  • 真正的业务规则、任务状态和续工记录，通常还要继续看 .trellis/spec/、.trellis/tasks/ 和 .trellis/workspace/。
必看 2

.trellis/tasks/：先看现在正在做什么

这里是任务区。当前 0.5.x 的"当前任务"主要由 .trellis/.runtime/sessions/ 里的会话指针决定， 手动查看时最好配合脚本确认，不要再只看旧式全局 .current-task 文件。

  • 用 task.py list 看哪个任务带有 current 标记。
  • 看任务目标和验收标准。
  • 看还有没有未完成的下一步。
必看 3

.trellis/workspace/：先看上次做到哪

这里通常是 journal。最适合回答"上一轮具体做了什么、改了哪些文件、下次该从哪继续"。

  • 中断后恢复上下文先看它。
  • 用户想知道进度时也先看它。
  • 它比聊天记录更可靠，因为它已经写回仓库。
必看 4

.trellis/spec/：正式开写前必须看规则

它不需要第一眼全读，但一旦准备写代码，就必须读相关 index 和 checklist 指向的具体规则文件。

  • 看代码风格、目录约定、质量标准。
  • 发现新规则时，也会更新回这里。
  • 它解决的是"怎么做才算符合这个项目"。
按工具再看

剩下这些位置，不是每个项目都有

位置 什么时候会出现 主要给谁看

.codex/ 只有你接入 Codex 时才会生成 Codex 项目级 config、agents 和 hooks。当前 .codex/hooks.json 注册的是 UserPromptSubmit workflow-state hook；Codex hooks 还需要用户级开启和审批。

.opencode/ 接入 OpenCode 时才会生成 OpenCode 的插件、命令和 agents 目录，和 Codex 不是一回事。
最适合小白的阅读顺序： 先看 AGENTS.md，再用 task.py list 或 get_context.py 确认当前任务，然后看 .trellis/workspace/ 最新 journal。只要准备改代码，就必须补读相关 .trellis/spec/。 仓库里的事实永远优先于 AI 自己的记忆。
Tools

先选接入方式，再看 Claude 和 Codex 怎么分别使用

两者都能读项目文件、改代码、跑命令，但入口格式、自动注入方式、项目目录并不一样。 你可以只用其中一个，不需要为了用 Trellis 强行两个一起装。 按当前生成文件，Claude 走 .claude/ 的 commands、hooks、agents、skills + /trellis:continue / /trellis:finish-work； Codex 走 AGENTS.md + .codex/ hooks/agents + .agents/skills/ 里的 $trellis-... skills。

Claude Code

命令入口是 Slash Commands

Claude Code 主要通过 .claude/commands/trellis/ 里的命令文件和 .claude/hooks/ 工作。 当前 0.5.x 项目实际生成的 slash commands 是 /trellis:continue 和 /trellis:finish-work； 需求梳理、编码前读规范、检查和更新 spec 主要由生成的 skills / agents 和 per-turn workflow-state 驱动。 如果项目把 AGENTS.md 定义成跨平台共享首页，Claude 也应该遵守并读取，但这属于项目层共享入口，不是 Claude 模板默认唯一主入口。

/trellis:continue /trellis:finish-work trellis-brainstorm trellis-check
  • Claude 的 SessionStart hook 会在启动、清空、压缩后注入上下文；通常不再需要手动跑旧版 /trellis:start。
  • Claude 使用 .claude/commands/、.claude/hooks/、.claude/agents/、.claude/skills/；不要把 Codex 的 .agents/skills/ 当成 Claude 主入口。
  • PreToolUse hook 会在派发 trellis-implement / trellis-check 之前注入 PRD、JSONL 上下文和相关 spec。
  • AGENTS.md 对 Claude 也有约束力；区别在于 Claude 的平台专属主入口仍然是 .claude/ 这套命令和 hooks。
  • 如果旧教程或旧源码里出现 /trellis:start、/trellis:record-session，以你项目里实际存在的 .claude/commands/trellis/ 文件为准。
Codex

命令入口是 $skill，不是 /slash

Codex 不能直接执行 Claude 那套 /trellis:start 或 /trellis:finish-work 命令。 它用的是 $trellis-start、$trellis-continue、$trellis-finish-work 这类带前缀的 skills， 并结合 AGENTS.md、.codex/ 与 .agents/skills/ 工作。

$trellis-start $trellis-continue $trellis-finish-work $trellis-brainstorm $trellis-before-dev $trellis-check
  • 可以直接说人话，但想显式跑流程时要用 $...，不是 /...；Codex 不会把 Claude 的 slash command 当成可执行技能。
  • 当前 Codex 既有 .codex/ 项目级 hooks/config/agents，也会写共享 .agents/skills/。
  • 这次生成的 .codex/hooks.json 只启用了 UserPromptSubmit；即使目录里有 .codex/hooks/session-start.py，也不要把它当成当前已注册的 Codex SessionStart 入口。
  • Codex hooks 需要用户级 [features].hooks = true、信任当前项目，并在 /hooks TUI 里批准 Trellis hook，workflow-state 才会自动注入。
  • Codex 默认 dispatch_mode 是 inline：主 Codex 会话直接加载 trellis-before-dev 后改代码，再用 trellis-check 检查；除非你显式把配置改成 sub-agent。

进阶玩法

并行开发：0.5.x 里先别当成日常入口

旧教程里常把 parallel 当成进阶入口，但你这次 0.5.15 生成的 Claude 命令目录里没有 parallel.md， Codex 也默认走 inline 模式。实际教学里更稳的是：先把需求拆成多个独立 Trellis 任务，一次只推进一个；确实需要并行时，再根据当前项目生成的脚本和平台能力单独设计。

你可以这样说

继续 Todo List 项目。我要做 3 件事：任务筛选、剩余数量统计、补基础测试。请先判断它们是否能拆成独立任务；如果会互相改同一批核心文件，请建议我先做哪一项。

Codex 侧现状

Codex 默认 dispatch_mode 是 inline，普通任务由主会话直接读规范、改代码、检查。不要把旧版 $parallel 或 Claude 旧命令套进 Codex。

什么时候适合用

适合多个小任务边界清楚、改动文件尽量少重叠的时候。 如果几件事都会改同一个核心文件，或者需求本身还没规划清楚，先走上面的第 2 步和第 7 步，把方案拆清楚，再决定要不要并行。

先记住这个原则：

并行最适合"可拆分、低冲突、范围窄"的任务，不适合把一个还没想清楚的大需求硬拆给多个 AI 同时乱改。

典型场景 适合并行开发 不适合并行开发
Todo List 第二轮优化 任务筛选、剩余数量统计、补测试，三件事边界清楚，且可以尽量分开改不同文件。 筛选逻辑、列表渲染、数据结构重构都同时改同一个核心状态文件，冲突会很多。
前后端联动需求 一个 agent 做前端展示，一个 agent 做独立测试或文档补充，接口已经先定义清楚。 接口协议还没定、字段还在变，这时并行只会让前后端各写各的。
需求清晰度 已经过第 2 步或第 7 步规划，PRD 和验收标准都比较明确。 需求还是一句"继续优化一下体验"，连先做哪块都没定，不该直接 parallel。
文件冲突风险 每个 worktree 任务都能控制在一小块范围内，改动尽量少重叠。 多个 agent 都要一起改同一批核心文件、同一组状态管理逻辑或同一条主流程。

几个最容易问到的问题

09
Q1. 必须在同一个项目里同时装 Claude 和 Codex 吗？
不需要。按这次生成结果，trellis init --claude 只会生成 Claude 入口，trellis init --codex 会生成 .codex/ 和共享 .agents/skills/。 你完全可以只用一个，也可以两个都接。
Q2. Trellis 是不是每天都要重新初始化一次？
不是。trellis init 是首次接入命令。项目已经接入后，日常只是进入项目目录，然后启动 claude 或 codex。
Q3. Claude 和 Codex 会自动共享所有记忆吗？
不会。真正共享的是项目里的事实层，例如 AGENTS.md 和 .trellis/。 但平台专属入口并不一样：Claude 默认主要看 .claude/，Codex 默认主要看 AGENTS.md 和 .agents/skills/。 模型私有 memory 不能直接当项目事实。
Q4. 我中间清空了对话，还能接着做吗？
可以。正确做法不是"让 AI 回忆昨天"，而是让它重建证据链：先读 AGENTS.md，再用 get_context.py 或 task.py list 确认当前任务，再看最新 journal。只要准备写代码，就必须补读相关 spec 和 workflow。
Q5. Claude 也要读 AGENTS.md 吗？
要分两层理解。Claude 的平台专属默认入口是 .claude/ 里的 slash commands 和 hooks； 但如果仓库里有 AGENTS.md，而且里面写了项目级指令或统一入口规则，Claude 也应该读，也应该遵守。 不要把它理解成"Claude 不读 AGENTS"，更准确的说法是"Claude 的平台入口不只靠 AGENTS，Codex 则更依赖 AGENTS 作为项目说明入口之一"。
Q6. Codex 一定要记住很多技能名吗？
不一定。项目已经接好 Trellis 时，很多场景下直接说自然语言就够了。 但如果你要显式触发流程，Codex 要用 $trellis-start、$trellis-continue、$trellis-check、$trellis-finish-work 这类带前缀的 skills， 不是 Claude 那种 /trellis:continue、/trellis:finish-work slash commands。
Q7. 为什么我下次进来时，AI 说不清当前项目进度？
常见原因是上一轮没有按 Phase 3 收尾：代码没提交、任务没归档、journal 没更新，或者当前会话指针没有正确指向任务。 0.5.x 里 session 记录已经并入 finish-work，所以收尾时要在工作提交之后运行 /trellis:finish-work 或 $trellis-finish-work。
Q8. 为什么初始化后总有一个 00-bootstrap-guidelines 任务？
这是 Trellis 首次初始化时自动创建的规范补齐任务，不是异常。它会提醒你先根据真实代码填写 .trellis/spec/，并通过 session 级运行时指针成为当前任务。 处理方式是：补好最小规范后执行 task.py archive "00-bootstrap-guidelines"；如果只是空项目演示，也可以明确暂时跳过并归档，再创建真正的业务任务。task.py finish 只清当前会话指针，不负责完成归档。
